## 문제

지금 Beacon은 산출물을 **존재 여부**로만 판정한다. `scanner.ts`/`process.ts`는 파일 내용을 읽지 않는다. 그래서 "README가 있다"까지만 알고, "README에 실행 방법이 빠졌다", "PRD에 비목표가 없다"는 모른다.

사용자 피드백: 각 산출물이 "이 파일이 부족하다"에서 끝나는 게 아니라, 무엇이 부족한지·무엇을 더하면 좋은지 가이드를 줬으면 한다.

## 왜 지금 할 수 있나 (인프라 준비됨)

- **문서 내용 읽기가 이미 가능**하다. v0.3.0에서 만든 `/api/file` + `readViewableFile`이 `.md/.mdx/.txt`를 안전하게(경로 이탈 차단) 읽어준다.
- 새 네트워크·의존성이 필요 없다.

## 원칙 부합 확인 (중요)

- **ADR 0002(외부 연동 안 함)와 충돌하지 않는다.** 이건 외부 SaaS 연동이 아니라 **로컬 파일을 더 깊게 읽는 것**이다. "파일·Git이 진실"을 오히려 강화한다.
- **ADR 0003(P0–P4 고정)과 무관하다.** 단계 기준을 바꾸는 게 아니라, 각 문서 안의 완성도를 본다.
- 제품 원칙 유지: 모든 신호는 **근거·출처·다음 행동**을 가진다. 문서 가이드도 "무엇이 없다(근거) → 이렇게 채워라(다음 행동)" 형태여야 한다.

## 접근 방향 (구조적, 규칙 기반)

내용을 "이해"하려 하지 말고(모델·외부 호출 없이), **문서 종류별 체크리스트를 구조적으로 점검**한다. 파일명·헤딩·섹션 존재를 규칙으로 확인하는 것이지, 문장의 의미를 판단하는 게 아니다. 이래야 빠르고, 오판이 적고, 로컬·오프라인 정체성을 지킨다.

예시 — 종류별 기대 섹션(초안, 실제 착수 때 확정):

| 종류 | 점검할 요소(헤딩/키워드 존재) |
|---|---|
| README (overview) | 프로젝트 목적 · 설치/실행 방법 · 사용법 |
| 기획 (planning) | 해결할 문제 · 목표/사용자 · MVP · 비목표 |
| 설계 (architecture) | 구성요소/구조 · 데이터 소유권 · 주요 결정 |
| 검증 (quality) | 대상 범위 · 방법 · 기대 결과 |
| 릴리스 (release) | 버전 · 변경점 · 설치/전달 |

각 문서에 대해: "있는 섹션 ✓ / 빠진 섹션 → 이렇게 채우면 좋아요" 를 보여준다.

## 구현 스케치 (단계별, 각 1 PR)

1. **core에 순수 함수 `assessDocument(kind, content)`** — 헤딩/키워드를 파싱해 "충족/누락 섹션 + 다음 행동"을 반환. 내용은 읽되 규칙으로만 판단. 테스트 먼저.
2. **서버**: `/api/file`가 이미 내용을 주므로, 대시보드가 문서 뷰어를 열 때 `assessDocument` 결과를 함께 보여주거나, 별도 경량 엔드포인트로 노출.
3. **대시보드**: 산출물 그룹의 각 파일 옆/뷰어 안에 "구성 점검"(빠진 섹션·다음 행동) 표시. 기존 문서 뷰어와 자연스럽게 연결.
4. (선택) Project Book에 문서 구성 점검 요약 포함.

## 유의점 / 리스크

- **오판을 만들지 말 것.** 헤딩이 없다고 "부족"으로 단정하면 자유형 문서에서 잘못된 신호가 난다. "없음"은 강한 판정이 아니라 "이런 섹션이 있으면 더 좋아요" 정도의 부드러운 제안으로.
- **판정(Health/Gate)과 분리.** 문서 내용 점검은 **가이드/제안**이지, Health 점수나 Gate 통과에 곧바로 반영하지 않는다(반영하려면 별도 논의). 지금 판정은 "존재"로 안정적이므로 흔들지 않는다.
- 종류별 기대 섹션은 팀마다 다를 수 있다 — 최소한의 보편 항목만 넣고, 과하게 규정하지 않는다.
- 바이너리 문서(pdf/docx 등)는 내용 점검 대상에서 제외(텍스트만).

## 다음 세션 첫 스텝

`assessDocument(kind, content)`의 계약(입출력)과 종류별 체크리스트 초안을 확정하고, core 순수 함수 + 테스트부터 착수한다.
