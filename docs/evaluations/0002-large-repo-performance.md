# External Evaluation 0002 — 대형 레포 성능·정확도 실측

| 항목 | 내용 |
|---|---|
| 대상 | `colinhacks/zod` (전체 히스토리 clone) |
| 규모 | 583 파일 · 2927 commit |
| 평가일 | 2026-07-22 |
| 기준 Beacon | `v0.2.0` (main, commit-author·gitignore·commit 상한 500 반영) |
| 목적 | 실무 규모 레포에서 스캔 속도·`.gitignore` 존중·분류 정확도 검증 |

## 방법

성숙한 실제 오픈소스(zod)를 전체 히스토리로 clone하고, 서버·의존성을 설치하지 않은 채 `scanProject`를 1회 실행해 측정했다.

## 결과

| 지표 | 값 | 판단 |
|---|---:|---|
| 스캔 시간 | **267 ms** | 🟢 실무 충분 |
| 관찰 파일 | 580 | 파일 상한(10000) 여유 |
| 발견 문서 | 46 (overview 4·release 2·document 40) | 분류 정확 |
| 읽은 commit | 500 | 상한에 도달 (아래 참고) |
| Health | 40% at_risk / 현재 P0 | 오탐 아님 (아래 참고) |

### 검증된 것

- **성능**: 583파일·2927commit을 0.3초에 관찰. 실무 규모에서 빠르다.
- **`.gitignore` 존중**: 빌드 산출물 `lib/`가 제외됨(무시 폴더를 산출물로 오인하지 않음).
- **분류 정확도**: monorepo의 패키지별 README를 overview로, `CHANGELOG`/`changelog.mdx`를 release로 정확히 분류.

## 발견 (실무 관점)

### 1. commit 상한 500이 대형 레포에선 여전히 창(window)이다
zod는 2927 commit인데 500만 읽는다. 판정 안정성(P4 릴리스 commit이 500 안에 있을 확률)에는 충분하나, "전 이력"은 아니다. 표시는 Timeline 30개·누적은 History로 제한되므로 UI 부담은 없다. 초대형 레포에서 판정이 흔들리면 상한을 더 올리거나 "릴리스/설계 관련 commit만 전 구간 스캔"하는 보강을 검토한다.

### 2. Health 40%는 오탐이 아니라 "적용 대상" 신호다
zod는 유명 라이브러리지만 README·문서(46개)만 있고, Beacon이 찾는 **기획(PRD/PRODUCT)·설계(ADR/architecture)** 형식 문서가 실제로 없다. 그래서 P0에서 멈춘다. 이는 정확한 관찰이다.

**핵심 결론: Beacon의 P0–P4는 "제품 개발 프로젝트" 기준이라, 라이브러리·오픈소스 저장소에는 그대로 맞지 않는다.** 정확도 결함이 아니라 적용 범위의 문제다. 라이브러리를 대상으로 쓰려면 기준(템플릿)이 달라야 하며, 이는 "팀별 P0–P4 커스터마이즈" 논의와 직접 연결된다.

## 다음 행동

- 대형 레포 판정 안정성이 문제되면 commit 상한/릴리스 commit 전구간 스캔 보강.
- "제품 프로젝트 vs 라이브러리"처럼 대상 성격이 다르면 기준 세트가 필요하다 — 커스터마이즈 결정(별도 ADR)에서 다룬다.
