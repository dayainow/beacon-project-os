# ADR 0003 — P0–P4를 고정으로 두고, 대상을 "제품 개발 프로젝트"로 명확히 한다

| 항목 | 내용 |
|---|---|
| 상태 | Accepted |
| 결정일 | 2026-07-22 |
| 관련 | [Evaluation 0002](../evaluations/0002-large-repo-performance.md), [PRODUCT.md](../PRODUCT.md), [ADR 0002](0002-no-external-integrations.md) |

## 배경

Evaluation 0002에서 zod(성숙한 라이브러리)를 스캔했더니 Health 40%·P0에서 멈췄다. 오탐이 아니라 정확한 관찰이었다 — zod엔 Beacon이 찾는 **기획(PRD/PRODUCT)·설계(ADR/architecture)** 형식 문서가 실제로 없다. 오픈소스 라이브러리는 그런 문서를 잘 쓰지 않기 때문이다.

여기서 두 선택지가 충돌했다.
- **팀/대상별로 P0–P4 기준을 커스터마이즈**하면 라이브러리에도 맞출 수 있다.
- 그러나 그러면 "Beacon이 정한 하나의 표준"이라는 단순함이 흐려진다.

## 결정

**P0–P4와 각 Gate 요구조건을 고정으로 유지한다. 대신 Beacon의 적용 대상이 "제품 개발 프로젝트"임을 문서·화면에서 명확히 한다.**

즉 커스터마이즈를 열지 않는다. 대상이 안 맞으면 기준을 바꾸는 게 아니라, "이 도구가 그 대상용이 아니다"라고 정직하게 밝힌다.

## 이유

1. **단순함이 제품의 힘이다.** "모든 프로젝트가 같은 P0–P4로 정직하게 비교된다"가 Beacon의 명료함이다. 팀마다 기준이 다르면 판정을 서로 신뢰·비교할 수 없고, 설정 자체가 유지보수·학습 부담이 된다.

2. **판정의 근거가 흔들리지 않는다.** 기준이 고정이면 "왜 이 판정인가"가 항상 같은 규칙에서 나온다(제품 원칙: 모든 신호는 근거·출처·다음 행동을 가진다). 커스터마이즈는 이 일관성을 깬다.

3. **문제는 "정확도"가 아니라 "적용 범위"였다.** zod 결과는 Beacon이 틀린 게 아니라, 애초에 라이브러리가 대상이 아니었을 뿐이다. 범위를 명확히 하면 오해가 사라진다.

이 결정은 [ADR 0002](0002-no-external-integrations.md)(외부 연동 안 함)와 같은 원칙을 따른다 — **제품을 넓히기보다 정체성을 선명하게 유지한다.**

## 대상 정의

Beacon은 **하나의 제품/서비스를 처음부터 완성까지 만들어 가는 프로젝트**에 맞다. P0 기획 → P1 설계 → P2 개발 → P3 검증 → P4 배포의 흐름을 따르는 프로젝트.

**잘 맞는 대상**: 새 제품·기능·서비스 개발 저장소(기획·설계 문서를 쓰는 팀).
**맞지 않는 대상**: 범용 라이브러리·프레임워크·유틸리티처럼 기획·설계 문서 없이 코드·README 중심으로 가는 저장소. Health가 낮게 나오는 것은 결함이 아니라 "대상이 아님"의 신호다.

## 제외 (하지 않는 것)

- `.beacon` 설정으로 P0–P4 단계·Gate 요구조건을 바꾸는 기능
- 대상 유형별(라이브러리/앱/모노레포 등) 별도 기준 세트
- 사용자 정의 단계 추가

## 결과

Beacon은 "제품 개발 프로젝트를 고정 표준(P0–P4)으로 정직하게 비추는 도구"라는 정체성을 유지한다. 라이브러리 등 대상이 다른 저장소에서는 낮은 Health가 정확한 신호이며, 이를 문서로 안내해 오해를 막는다. 향후 라이브러리 지원이 진짜 필요해지면, 그때는 "커스터마이즈"가 아니라 "별도 대상 유형" 결정으로 새 ADR에서 다룬다.
