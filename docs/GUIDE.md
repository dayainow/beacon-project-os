# Beacon 사용 가이드

Beacon은 프로젝트 폴더를 읽어 "지금까지 무엇이 되었고, 무엇이 빠졌으며, 언제 무슨 작업을 했는지"를 보여주는 로컬 도구다. 파일과 Git만 관찰하며, 아무것도 실행하지 않는다.

손으로 정할 것은 하나뿐이다 — 이번에 완성할 목표(마일스톤). 나머지는 파일과 Git에서 자동으로 채워진다.

- 적용 대상: 하나의 제품·서비스를 처음부터 완성까지 만들어 가는 프로젝트. (라이브러리·유틸리티는 대상이 아니다 — [PRODUCT.md](PRODUCT.md) 참고)
- 데이터: 모두 내 컴퓨터의 `.beacon/` 폴더에만 저장되고, 외부로 전송되지 않는다.
- 요구사항: Node.js 22.13 이상.

---

## 빠른 시작

확인할 프로젝트 폴더에서 아래 세 줄이면 충분하다.

```bash
pnpm add --save-dev @dayainow/beacon   # 설치
pnpm exec beacon init                  # 붙이기
pnpm exec beacon open                  # 대시보드 열기
```

브라우저에 대시보드가 열린다. 아래 가이드에서는 짧게 `beacon`으로 쓴다.

---

## 단계별 흐름

### 1. 붙이기 — `beacon init`

폴더에 `.beacon/` 인덱스를 만든다. 원본 파일은 건드리지 않는다. Git 저장소면 변경 이력까지, 아니어도 파일은 읽는다.

### 2. 열어보기 — `beacon open`

`127.0.0.1:4300`에 대시보드가 열린다. 왼쪽 메뉴로 현재 상태를 본다.

| 화면 | 답하는 질문 |
|---|---|
| 개요 | 지금 프로젝트가 건강한가? (진행 단계 도넛, Health, 활동량, 점검 목록) |
| 단계 | 기획→배포 중 어디까지 왔나? (P0–P4, 각 단계 클릭하면 상세) |
| 산출물 | 어떤 문서가 있나? (종류별로 묶어서, 텍스트 문서는 클릭해 내용 보기) |
| 히스토리 | 지금까지 뭘 했나? (일자별·Cycle·상세 세 탭) |
| 가이드 | 처음이라면 여기부터 |

대시보드 오른쪽 위 "자동 스캔"을 켜면 15초마다 상태가 자동으로 갱신된다.

### 3. 마일스톤 시작 — `beacon cycle start`

여기가 사람이 정하는 유일한 부분이다. 마일스톤(Cycle)은 완성되는 목표 하나다.

```bash
beacon cycle start "대출 기능 MVP" --goal "대출·반납 첫 흐름을 완성한다"
```

- 목표는 "이게 되면 끝"이라고 판정할 수 있게 쓴다. (좋음: "결제 연동을 통과한다" / 애매함: "결제 관련 작업")
- 크기는 며칠~2주. 너무 크면 나누고, 한 커밋짜리면 합친다.
- 시작하는 순간의 상태가 출발점(기준선)으로 기록된다.

### 4. 작업하기

Beacon에는 아무것도 입력하지 않는다. 평소처럼 코드를 쓰고 커밋하면, 자동 스캔을 켠 채로 히스토리에 저절로 쌓인다. 이슈·태스크를 손으로 쪼개 넣을 필요가 없다 — 커밋 하나, 문서 하나가 곧 기록이다.

### 5. 마일스톤 종료 — `beacon cycle complete`

목표 결과물이 나오면 닫는다.

```bash
beacon cycle complete --summary "대출·반납 첫 흐름 완성"
```

출발점과 종료 시점을 비교한 성과(산출물 변화, 커밋 수, Health·단계 이동)가 자동으로 요약되어 히스토리 Cycle 탭에 남는다. 그다음 새 목표를 `cycle start`로 이어가면 된다.

### 6. 내보내기(선택) — `beacon export`

지금까지의 상태를 하나의 공유용 문서(`PROJECT_BOOK.md`)로 뽑는다. 프로젝트 정체성·현재 상태·P0–P4·마일스톤 로그를 담으며, 내 컴퓨터의 절대 경로는 포함하지 않는다.

## 화면 읽는 법

### Health — 다섯 가지 기준

Health는 다섯 가지 중 몇 개가 준비됐나의 비율이다.

| 기준 | 통과 조건 |
|---|---|
| 소개 | 루트 README |
| 기획 | PRD·PRODUCT·요구사항 문서 |
| 설계 | ADR·Architecture·Design 문서 |
| Git | Git 저장소인가 |
| 이력 | 커밋이 있나 |

다섯 개 다 준비되면 순조로움, 하나라도 빠지면 확인 필요, 절반 이하면 근거 부족으로 표시된다. 하나 빠졌다고 낮게 나오는 것은 의도된 동작이다 — 무엇이 빠졌는지 정직하게 드러내기 위한 것이지, 프로젝트가 나쁘다는 뜻이 아니다.

Health(문서 기초가 갖춰졌나)와 단계(기획→배포 중 지금 어디)는 겹치지만 목적이 다르다.

### 산출물 — 무엇을 문서로 잡나

- 문서로 인식하는 확장자: `.md .mdx .txt .pdf .doc .docx .ppt .pptx .xls .xlsx` (Markdown뿐 아니라 PDF·Word·PPT·Excel도 잡는다)
- 소스 코드(`.ts .py .go` 등)는 문서가 아니라 "소스 파일 수"로 세어 개발 단계 판정에 쓴다.
- `.gitignore`에 넣은 파일·폴더와 도구 설정(`.claude` `.github` `skills` 등)은 판정에서 제외된다.
- 종류는 파일명·경로의 단어로 자동 분류한다(한국어도 인식: 기획안·설계서·요구사항서 등).

### 히스토리 — 세 가지 탭

- 일자별: 날짜별로 무엇을 몇 건 했는지. 타임라인 형태로 커밋·문서 변경을 시간순으로 본다.
- Cycle: 마일스톤 로그. 각 Cycle을 펼치면 그 기간의 커밋(이슈), 커밋을 펼치면 바뀐 파일(태스크)까지 계층으로 본다.
- 상세: 커밋·문서 변경을 하나씩, 스캔 사이에 바뀐 것을 함께 본다. 커밋에는 작성자가 표시된다.

## 명령어 요약

| 명령 | 역할 |
|---|---|
| `beacon init` | 현재 폴더에 `.beacon/`을 만들고 초기화 |
| `beacon open` | 로컬 대시보드 열기 (`--port`, `--no-browser`) |
| `beacon identity` | 프로젝트 이름·경로·Git 기준점 확인 |
| `beacon cycle start "이름" --goal "목표"` | 마일스톤 시작 |
| `beacon cycle complete [--summary "요약"]` | 마일스톤 종료 |
| `beacon cycle status` | 현재 Cycle 확인 |
| `beacon export [--output 경로]` | `PROJECT_BOOK.md` 생성 |

모든 명령은 현재 폴더를 기본 대상으로 하며 `--root 경로`로 다른 폴더를 지정할 수 있다.

## 자주 묻는 것

**프로젝트가 끝나야 쓰는가.** 아니다. 진행 중에 쓴다. 대시보드를 열어두고 자동 스캔을 켜면 작업하는 대로 쌓인다. `export`는 언제든 뽑는 스냅샷일 뿐 필수가 아니다.

**마일스톤을 꼭 종료해야 하는가.** 아니다. 시작만 하고 진행 중으로 둬도 된다. 종료는 다음 목표로 넘어갈 때만 한다.

**내 코드가 외부로 나가는가.** 아니다. 파일은 전송되지 않고, Beacon 데이터는 재생성 가능한 `.beacon/` 폴더에만 저장된다.

**라이브러리에서 Health가 낮게 나온다.** 결함이 아니다. Beacon은 제품 개발 프로젝트용이라, 기획·설계 문서를 쓰지 않는 라이브러리에는 맞지 않는다. 낮은 Health는 "대상이 아님"의 정직한 신호다.
